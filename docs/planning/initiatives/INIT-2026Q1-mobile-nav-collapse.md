# INIT-2026Q1-mobile-nav-collapse

## Название
Мобильная навигация: скрывать боковое меню в compact menu на телефонах

## Цель
На мобильных экранах основная навигация не должна занимать первый экран. Пользователь должен видеть контент household сразу после верхней панели, а навигация должна открываться через compact menu: burger / три точки / drawer.

## Контекст
Сейчас на desktop левое меню работает приемлемо: слева навигация, справа контент задач и покупок.

На mobile меню превращается в большой вертикальный блок перед контентом:
- занимает почти весь первый экран;
- отодвигает задачи ниже;
- ухудшает базовый сценарий “открыл household → увидел задачи”.

## Scope: In
- На мобильной ширине скрыть левую навигацию из основного потока.
- Добавить compact menu control в header: burger или три точки.
- По нажатию открывать навигацию как drawer / overlay / dropdown.
- Сохранить существующие пункты меню:
  - Задачи
  - Рутины
  - Аналитика
  - Прогресс
  - Зоны
  - Уведомления
  - Участники
  - Пригласить участника
- Сохранить desktop layout без регрессии.
- Добавить/обновить минимальные UI-тесты или screenshot/manual checklist, если автотестов UI пока нет.

## Scope: Out
- Не менять backend API.
- Не менять household/task/shopping доменную модель.
- Не менять авторизацию.
- Не редизайнить всю страницу.
- Не менять desktop-навигацию без необходимости.
- Не вводить полноценную дизайн-систему.

## Acceptance Criteria
1. На desktop ширине навигация остаётся слева.
2. На mobile ширине навигация не занимает верх страницы перед контентом.
3. На mobile есть видимый control для открытия меню.
4. При открытии меню доступны все текущие пункты навигации.
5. Меню можно закрыть:
   - повторным нажатием;
   - кликом вне меню, если используется overlay/drawer;
   - клавишей Escape, если проект поддерживает keyboard handling.
6. После выбора пункта меню mobile menu закрывается.
7. Контент задач виден без долгой прокрутки сразу после header или near-first-screen.
8. Нет горизонтального скролла.
9. Desktop layout не сломан.
10. Изменение не затрагивает backend contracts/API.

## Risk
Low/Medium.
Низкий риск по доменной логике, но есть риск сломать responsive layout/header.
