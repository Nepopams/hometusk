# ADR-002: MVP сценарий "Текстовая команда"

**Статус:** Принято
**Дата:** 2026-01-11
**Авторы:** Product Team

---

## Контекст

MVP фокусируется на валидации ключевой гипотезы:

> Пользователи могут создавать и назначать домашние задачи через естественный язык, без изучения структурированного интерфейса.

Голосовой ввод (описанный в ADR-001) отложен на будущие версии. MVP использует только текстовый ввод.

---

## Решение

### Принцип UX

**Цель:** за 3–5 секунд превратить текст в назначенную задачу.

**Золотое правило MVP:**
> Текст → AI-решение → 1 экран подтверждения → готово.

**Критерий успеха:** пользователь не думает, как правильно "оформить" задачу.

---

## User Journey

### Шаг 1: Ввод команды

**UX:**
- Текстовое поле на главном экране (web/mobile)
- Placeholder: "Что нужно сделать?"
- Отправка по Enter или кнопке "→"

**Примеры команд:**
- "Убрать кухню сегодня вечером"
- "Маша, купи молоко"
- "Нужно помыть машину до субботы"

### Шаг 2: AI-обработка

**Системно:**
```
command_received
    ↓
intent_resolved (что: убрать, где: кухня)
    ↓
context_enriched (участники дома, их доступность)
    ↓
decision_made (кому: Мария, когда: сегодня 18:00-22:00)
```

**UX:**
- Индикатор загрузки (1–2 сек)
- Никаких промежуточных вопросов

### Шаг 3: Подтверждение

**UX:**
Карточка с результатом:

```
┌─────────────────────────────┐
│  Создать задачу?            │
├─────────────────────────────┤
│  Что: Убрать кухню          │
│  Кому: Мария                │
│  Когда: Сегодня вечером     │
├─────────────────────────────┤
│  [Создать]  [Изменить]      │
└─────────────────────────────┘
```

**Подсказка:** "Можно исправить перед созданием"

### Шаг 4: Результат

**UX:**
- "Готово. Задача создана."
- Кнопки: "Открыть задачу", "Создать ещё"

**Системно:**
- `task_created` event
- `task_assigned` event
- Push-уведомление исполнителю

---

## Событийная модель

```
command_received
    ↓
intent_resolved
    ↓
context_enriched
    ↓
decision_made
    ↓
decision_validated
    ↓
task_created
    ↓
task_assigned
    ↓
notification_sent
```

Каждое событие логируется в `DecisionLog` для аудита и отладки.

## Fallback-сценарии

### AI не уверен в решении

**Условие:** confidence < threshold (например, 0.7)

**Решение:**
- Показать интерпретацию с пометкой "Проверьте"
- Подсветить неуверенные поля
- Разрешить редактирование перед созданием

### Нет участников в доме

**Решение:**
- Назначить задачу себе
- Показать: "Назначено вам. Пригласите людей, чтобы поручать им."

### AI недоступен

**Решение:**
- Использовать базовые эвристики:
  - Назначить инициатору
  - Без дедлайна
- Пометить задачу как `created_via_fallback`
- Показать: "Создано в упрощённом режиме"

### Неоднозначный адресат

**Примеры:** "мы", "кто-нибудь", "надо бы"

**Решение:**
- Назначить инициатору ("мне")
- В карточке показать возможность смены исполнителя

## Набор экранов (MVP)

| # | Экран | Назначение |
|---|-------|------------|
| 1 | Home | Список задач + поле ввода команды |
| 2 | Command Input | Модальное окно / inline ввод |
| 3 | Confirm Task | Карточка подтверждения |
| 4 | Task Detail | Детали задачи |
| 5 | Task List | Список задач (фильтры: мои, все) |
| 6 | Invite Members | Приглашение участников |

## Сравнение с ADR-001

| Аспект | ADR-001 (Voice) | ADR-002 (Text) |
|--------|-----------------|----------------|
| Ввод | Голос | Текст |
| Транскрипция | Да | Нет |
| Entry points | Микрофон, виджет, lock screen | Текстовое поле |
| Время до создания | 5–10 сек | 3–5 сек |
| Сложность | Высокая (STT, шум) | Низкая |
| MVP scope | Нет | Да |

ADR-001 остаётся актуальным для будущих версий с голосовым вводом.

## Технические требования

### Обязательные для MVP

- LLM для интерпретации команд
- Schema validation для AI output
- Decision logging
- Push notifications
- Degraded mode (heuristics fallback)

### Не требуется для MVP

- Speech-to-Text
- Real-time голосовой streaming
- Wake word detection
- Audio level visualization

---

## Последствия

### Положительные

- Минимальная сложность для MVP
- Быстрая валидация гипотезы
- Проще тестировать (текст vs аудио)
- Меньше edge cases (нет шума, акцентов)

### Риски

- Текст менее "магичен" чем голос
- Может быть менее удобен в быту (руки заняты)
- Требует клавиатуру / экран

### Митигация

- Голосовой ввод — следующая итерация (ADR-001)
- MVP фокусируется на ценности AI-решений, не на модальности ввода
