# ADR-001: MVP сценарий "Голосовое создание задачи"

**Статус:** Принято
**Дата:** 2026-01-11
**Авторы:** Product Team

## Контекст

Для MVP продукта необходимо определить ключевой пользовательский сценарий, который продемонстрирует ценность приложения и позволит быстро валидировать гипотезы. Выбран сценарий голосового создания и назначения задач в рамках домохозяйства.

## Решение

### Принцип UX

**Цель:** за 5–10 секунд превратить фразу в назначенную задачу без диалогов.

**Критерий успеха:** пользователь не думает, как правильно "оформить" задачу.

**Золотое правило MVP:**
> Голос → транскрипт → 1 экран подтверждения → готово.
> Никаких "давайте уточним 6 параметров".

---

## 1. Персоны и роли

| Роль | Описание |
|------|----------|
| **Инициатор** | Тот, кто сказал команду. Обычно "дом-менеджер", но может быть любой участник |
| **Исполнитель** | Член дома/семьи/сосед. В B2C — приглашённый контакт |

---

## 2. Пререквизиты (минимальные)

### Первое открытие (30–60 секунд, без анкеты)

**Экран 1: "Создать дом"**
- Поле: "Название (необязательно)" → по умолчанию "Дом"
- Кнопка: "Продолжить"

**Экран 2: "Кто живёт/участвует?"**
- Кнопка: "Пока один" (быстрый старт)
- Кнопка: "Пригласить" (контакты/ссылка)

> **Важно:** без приглашений сценарий всё равно работает (назначаем себе). Это снижает входной порог.

---

## 3. Entry Points

### Mobile (must-have)
- Большая кнопка микрофона на главном экране (внизу по центру)
- Быстрый "+" (текстовый ввод как fallback)

### Lock screen/виджет (nice-to-have)
- "Задать команду" → сразу запись голоса

### Web (MVP)
- Заточен под обзор + аналитика + ручные правки
- Без голосового ввода в MVP

---

## 4. Основной User Journey

### Шаг A: Запуск голосового ввода

**UX:**
- Пользователь тапает микрофон
- Открывается "Voice capture sheet" (нижняя панель на 60–70% экрана):
  - Большой круг "Запись"
  - Подсказка: "Говорите нормально, как человеку"
  - Кнопка "Отмена"

**Фидбек:**
- Анимация уровня громкости
- Таймер (макс 10–12 сек)

**Системно:**
- Запрос разрешения на микрофон (один раз)
- Создание `voice_session_id`

**Fallback:**
- Если микрофон не разрешён → текстовое поле "Введите команду"

---

### Шаг B: Транскрипция и черновик

**UX:**
После записи показываем экран "Мы вас поняли так:":
- Крупно: распознанный текст (редактируемый)
- Кнопки: "Исправить", "Повторить запись"

**Системно:**
```
audio → transcript → Task Draft
```

Task Draft содержит:
- `title`: "Убрать кухню"
- `due`: "сегодня вечером" → интервал (18:00–22:00)
- `zone/tag`: "Кухня"
- `priority`: normal
- `suggested_assignee`: "Мария" (или "я")

**Критическая UX-идея:**
Пользователь видит человеческую карточку, а не структуру:

```
┌─────────────────────────────┐
│ Задача: Убрать кухню        │
│ Когда: сегодня вечером      │
│ Кому: Мария                 │
│                             │
│ [Создать]  [Изменить]       │
└─────────────────────────────┘
```

---

### Шаг C: Экран подтверждения

Один экран/боттом-шит "Подтвердите действие":

**UI:**
- Заголовок: "Создать поручение"
- Карточка:
  - **Что:** "Убрать кухню" (тап → редактирование)
  - **Кому:** "Мария" (тап → список людей)
  - **Когда:** "Сегодня вечером" (тап → Сегодня / Завтра / Дата / Без срока)
- CTA: `[Создать]` (primary), `[Отмена]` (secondary)
- Подсказка: "Пока учусь — можно править перед созданием"

**Системно:**
- Никаких диалогов
- Нажали "Создать" → фиксируем задачу

**Fallbacks:**
| Ситуация | Решение |
|----------|---------|
| Не уверен "кому" | Ставим "Мне" + "Назначено вам. Можно сменить исполнителя." |
| Не понял "когда" | "Без срока" |

---

### Шаг D: Создание и уведомления

**UX:**
Моментальный экран успеха (2–3 секунды):
- "Готово. Марии отправлено."
- Кнопки: "Открыть задачу", "Создать ещё"

**Системно:**
- Создание `task_id`
- Push исполнителю: "Поручение: Убрать кухню — сегодня вечером"
- Кнопки в push: "Принять", "Не могу", "Открыть"

**Модель принятия (MVP):**
> По умолчанию задача назначена. Исполнитель может нажать "Не могу" → инициатору прилетает запрос на переназначение.

---

## 5. Journey исполнителя

### Шаг E: Получение поручения

**UX:**
Push → карточка задачи:
- Что сделать
- Дедлайн
- Кнопки: `[Начать]`, `[Сделано]`, `[Не могу]`

**Статусы:**
```
assigned → in_progress → done
         ↘ declined
```

---

## 6. Unhappy Paths

### 6.1 Шум / плохая транскрипция
- Всегда показываем распознанный текст
- Даём быстро исправить и "повторить"
- **Не делаем:** автосоздание без подтверждения

### 6.2 Множественные адресаты ("мы", "кто-нибудь")
- Назначаем инициатору ("мне")
- В карточке показываем "Кому: мне"

### 6.3 Нет участников дома
- "Назначено вам. Пригласите людей, чтобы поручать им."

### 6.4 Исполнитель отказался
- Push инициатору: "Мария не может: 'не дома'"
- Кнопки: `[Переназначить]`, `[Снять задачу]`

---

## 7. Голосовой интерфейс

### MVP (рекомендовано)
**Push-to-talk:** нажал — говоришь — отпустил/стоп

| Плюсы | Минусы |
|-------|--------|
| Контроль | Не "как умная колонка" |
| Приватность | |
| Меньше ложных срабатываний | |

### v1.5 (альтернатива)
Голосовые сообщения в чат (как мессенджер)

### v2 (будущее)
Wake word / always listening — не для MVP (приватность, батарейка, юридика)

---

## 8. Микрокопи

| Вместо | Используем |
|--------|------------|
| "Я не понял" | "Похоже, вы имели в виду…" |
| "Уточните параметры" | "Можно уточнить, но не обязательно" |
| После создания | "Готово. Отправлено исполнителю." |

**Тон:** спокойный, без шуток "про быт"

---

## 9. Набор экранов (MVP)

1. **Home** — лента задач + кнопка микрофона
2. **Voice capture sheet** — запись голоса
3. **Transcript review** — редактируемый текст
4. **Confirm task** — что/кому/когда + Create
5. **Task detail** — детали задачи
6. **Executor inbox** — входящие исполнителя
7. **Invite members** — приглашение участников

---

## 10. Событийная модель

```
voice_session_started
    ↓
voice_transcript_ready
    ↓
task_draft_ready
    ↓
task_confirmed
    ↓
task_created
    ↓
task_assigned
    ↓
task_declined / task_done
```

> Это обеспечит наблюдаемость и пригодится для обучения/улучшения ИИ.

---

## Последствия

### Положительные
- Минимальный onboarding (30–60 секунд)
- Работает даже без приглашённых участников
- Простая ментальная модель для пользователя
- Готовая событийная архитектура для аналитики и ML

### Риски
- Качество распознавания речи критично для UX
- Необходима работа с edge cases (шум, акценты)
- Push-уведомления требуют настройки на устройстве

### Технические требования
- Speech-to-Text сервис
- LLM для интерпретации команд
- Push notification infrastructure
- Real-time статусы задач
