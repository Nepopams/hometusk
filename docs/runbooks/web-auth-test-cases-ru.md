# Тест-кейсы аутентификации веб-клиента

Ручные тест-кейсы для EP-004: Keycloak OIDC интеграция.

## Предварительные требования

- [ ] Keycloak запущен на `http://localhost:8180`
- [ ] Клиент `hometusk-web` настроен (см. `keycloak-setup-ru.md`)
- [ ] Backend запущен на `http://localhost:8080`
- [ ] Веб-клиент запущен на `http://localhost:5173`
- [ ] Регистрация пользователей включена в Keycloak realm

---

## TC-001: Регистрация нового пользователя

**Цель:** Проверить, что новый пользователь может зарегистрироваться и войти в приложение.

### Предусловия
- Нет существующего пользователя с тестовым email в Keycloak
- `VITE_AUTH_PROVIDER=keycloak` в `.env.local`

### Шаги

| Шаг | Действие | Ожидаемый результат |
|-----|----------|---------------------|
| 1 | Открыть `http://localhost:5173` | Редирект на `/login` |
| 2 | Осмотреть страницу входа | Видны кнопка "Sign in" и ссылка "Sign in to register" |
| 3 | Нажать "Sign in to register" | Редирект на страницу входа Keycloak |
| 4 | Нажать ссылку "Register" на странице Keycloak | Появляется форма регистрации |
| 5 | Заполнить форму регистрации (логин, email, пароль) | Форма принимает ввод |
| 6 | Отправить форму регистрации | Keycloak обрабатывает регистрацию |
| 7 | Наблюдать редирект | Редирект на `/callback`, затем на `/households` |
| 8 | Проверить страницу households | Отображается пустой список домохозяйств (Fix D) |

### Верификация
- [ ] Пользователь создан в Keycloak (Админ-консоль → Users)
- [ ] GET `/users/me` возвращает профиль нового пользователя
- [ ] Нет ошибок в консоли

---

## TC-002: Вход существующего пользователя

**Цель:** Проверить, что существующий пользователь может войти и получить доступ к households/tasks.

### Предусловия
- Тестовый пользователь существует (alice/alice123)
- У пользователя есть минимум одно домохозяйство (опционально)

### Шаги

| Шаг | Действие | Ожидаемый результат |
|-----|----------|---------------------|
| 1 | Открыть `http://localhost:5173/login` | Отображается страница входа |
| 2 | Нажать "Sign in" | Редирект на вход Keycloak |
| 3 | Ввести учётные данные (alice/alice123) | Данные приняты |
| 4 | Отправить форму входа | Keycloak аутентифицирует |
| 5 | Наблюдать редирект | Редирект на `/callback`, затем на `/households` |
| 6 | Проверить страницу households | Отображаются домохозяйства пользователя |
| 7 | Выбрать домохозяйство | Переход на `/households/{id}` |
| 8 | Нажать "Tasks" | Отображается список задач |

### Верификация
- [ ] Сессия пользователя активна
- [ ] Токен в sessionStorage (через oidc-client-ts)
- [ ] API-запросы включают заголовок Authorization

## TC-003: Сохранение сессии (обновление страницы)

**Цель:** Проверить, что сессия сохраняется при обновлении страницы.

### Предусловия
- Пользователь вошёл через Keycloak

### Шаги

| Шаг | Действие | Ожидаемый результат |
|-----|----------|---------------------|
| 1 | Войти как существующий пользователь | На странице `/households` |
| 2 | Нажать F5 (обновить страницу) | Страница перезагружается |
| 3 | Наблюдать состояние | Всё ещё на `/households`, всё ещё аутентифицирован |
| 4 | Перейти к задачам | Задачи загружаются успешно |

### Верификация
- [ ] Нет редиректа на `/login` после обновления
- [ ] Токен получен из sessionStorage
- [ ] API-запросы работают после обновления

## TC-004: Выход из системы

**Цель:** Проверить, что выход очищает сессию.

### Предусловия
- Пользователь вошёл через Keycloak

### Шаги

| Шаг | Действие | Ожидаемый результат |
|-----|----------|---------------------|
| 1 | Войти как существующий пользователь | На странице `/households` |
| 2 | Нажать Logout (если доступно) или очистить сессию вручную | Инициирован выход |
| 3 | Наблюдать редирект | Редирект на `/login` |
| 4 | Проверить sessionStorage | Данные OIDC очищены |
| 5 | Открыть `/households` напрямую | Редирект на `/login` |

### Верификация
- [ ] Сессия очищена из sessionStorage
- [ ] Защищённые маршруты редиректят на login
- [ ] Нет остаточного состояния аутентификации

## TC-005: Автоматический выход при 401

**Цель:** Проверить, что ответ 401 вызывает автоматический выход с сообщением об ошибке.

### Предусловия
- Пользователь вошёл через Keycloak
- Способ инвалидировать токен (дождаться истечения или вручную)

### Шаги

| Шаг | Действие | Ожидаемый результат |
|-----|----------|---------------------|
| 1 | Войти как существующий пользователь | На странице `/households` |
| 2 | Инвалидировать токен (очистить sessionStorage или дождаться истечения) | Токен недействителен |
| 3 | Навигировать для вызова API (например, обновить задачи) | API возвращает 401 |
| 4 | Наблюдать поведение | Происходит автоматический выход |
| 5 | Проверить URL | Редирект на `/login?error=session_expired` |
| 6 | Проверить страницу входа | Отображается сообщение "Session expired" |

### Верификация
- [ ] 401 вызывает handleAuthError
- [ ] Параметр error в URL
- [ ] Сообщение об ошибке видно на странице входа

---

## TC-006: Автоматическое обновление токена

**Цель:** Проверить, что токены обновляются автоматически до истечения.

### Предусловия
- Пользователь вошёл через Keycloak
- Короткое время жизни токена настроено в Keycloak (для тестирования)
- Scope `offline_access` включён

### Шаги

| Шаг | Действие | Ожидаемый результат |
|-----|----------|---------------------|
| 1 | Войти как существующий пользователь | На странице `/households` |
| 2 | Открыть DevTools браузера → Network | Мониторинг запросов |
| 3 | Дождаться приближения истечения токена | Срабатывает automaticSilentRenew |
| 4 | Наблюдать сеть | Отправлен запрос обновления токена |
| 5 | Продолжить использование приложения | Без прерываний, используется новый токен |

### Верификация
- [ ] В консоли видно "[Auth] Token expiring"
- [ ] Запрос silent renew во вкладке Network
- [ ] Нет редиректа на login

## TC-007: Режим разработчика (вставка токена)

**Цель:** Проверить, что режим разработчика работает без Keycloak.

### Предусловия
- `VITE_AUTH_PROVIDER=dev` в `.env.local`
- Получен валидный JWT токен (см. `local-dev.md`)

### Шаги

| Шаг | Действие | Ожидаемый результат |
|-----|----------|---------------------|
| 1 | Перезапустить dev-сервер в режиме dev | Сервер запускается |
| 2 | Открыть `http://localhost:5173/login` | Отображается форма вставки токена |
| 3 | Получить JWT токен через curl | Токен получен |
| 4 | Вставить токен в форму | Токен принят |
| 5 | Нажать Login | Попытка аутентификации |
| 6 | Наблюдать редирект | Редирект на `/households` |

### Верификация
- [ ] Видна форма вставки токена (не кнопки Keycloak)
- [ ] Токен сохранён в localStorage
- [ ] API-запросы работают с вставленным токеном

## TC-008: Некорректная конфигурация OIDC

**Цель:** Проверить корректную обработку некорректной конфигурации OIDC.

### Предусловия
- Некорректный `VITE_OIDC_AUTHORITY` (например, неправильный порт)

### Шаги

| Шаг | Действие | Ожидаемый результат |
|-----|----------|---------------------|
| 1 | Установить некорректный VITE_OIDC_AUTHORITY | Конфигурация некорректна |
| 2 | Перезапустить dev-сервер | Сервер запускается |
| 3 | Открыть `/login` | Страница входа отображается |
| 4 | Нажать "Sign in" | Попытка редиректа |
| 5 | Наблюдать поведение | Ошибка логируется, приложение не падает |
| 6 | Проверить консоль | "[OIDC] Missing required environment variables" или подобное |

### Верификация
- [ ] Приложение не падает
- [ ] Ошибка логируется в консоль
- [ ] Пользователь видит обратную связь (редирект корректно не срабатывает)

## TC-009: Проверка PKCE

**Цель:** Проверить, что PKCE используется в OIDC-потоке.

### Предусловия
- Пользователь не вошёл
- DevTools браузера открыты

### Шаги

| Шаг | Действие | Ожидаемый результат |
|-----|----------|---------------------|
| 1 | Открыть `/login` | Страница входа отображается |
| 2 | Открыть DevTools → Network | Мониторинг запросов |
| 3 | Нажать "Sign in" | Редирект на Keycloak |
| 4 | Изучить URL редиректа | Проверить query-параметры |

### Верификация
- [ ] URL содержит параметр `code_challenge`
- [ ] URL содержит `code_challenge_method=S256`
- [ ] Нет client_secret в URL (публичный клиент)

## TC-010: Отображение сообщений об ошибках

**Цель:** Проверить, что все типы ошибок отображаются корректно.

### Шаги

| Тип ошибки | URL | Ожидаемое сообщение |
|------------|-----|---------------------|
| Сессия истекла | `/login?error=session_expired` | "Session expired, please login again." |
| Сервис недоступен | `/login?error=auth_unavailable` | "Authentication service unavailable." |
| Ошибка аутентификации | `/login?error=auth_failed` | "Authentication failed. Please try again." |

### Верификация
- [ ] Каждый тип ошибки показывает корректное сообщение
- [ ] Стилизация ошибки видна (красный фон)
- [ ] Ошибка очищается при новой попытке входа

- [ ] Веб: `http://localhost:5173` загружается
